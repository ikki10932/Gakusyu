# 起動時の呼出履歴レビュー報告：ルータ設定機能の初期化フロー

本資料は、アプリ起動時におけるルータ設定関連処理の関数呼出履歴（詳細版）を構造化し、長期保守性の観点からレビューした結果をまとめたものです。将来の保守担当者が、初期化プロセスの依存関係と実行順序を即座に把握できることを目的としています。

## 1. 対象資料のタイトルと資料内容説明
**ルータ設定起動時の処理の呼び出し履歴（詳細解析版）**
アプリ起動から `CDkRouter1200::Analize` によるプロトコル別解析（IP, PPPOE, NAT等）に至るまでの完全な関数呼出シーケンスを記録したデバッグ資料。

## 2. 対象資料の改善の方向性
現状の呼出履歴は、実行順序を追うには適していますが、**「なぜその順序で呼ばれる必要があるのか」**という設計意図が見えにくい状態です。今後は、MFCのライフサイクル（ドキュメント/ビュー構造）と、特定のルータ機種（CDkRouter1220等）への依存関係を分離して記述し、機種追加時の影響範囲を特定しやすくする方向で整理を推奨します。特に `Analize` 内部の順序依存性（例：IPアドレス確定後にNAT設定）を明文化すべきです。

## 3. 対象資料の問題点（長期閲覧保守観点）
*   **暗黙的な依存関係**: `CDkRouter` 系列のコンストラクタ呼出タイミングと、実際の解析開始（`Analize`）の間に多くのUI初期化処理が挟まっており、データ準備完了の定義が曖昧です。
*   **解析ロジックのブラックボックス化**: `AnalizeRouterType` で決定された解析ルールが、具体的にどのように `Analize` 内の `ProcXXX`（`ProcIP`, `ProcPPPOE` 等）の実行順序に影響を与えているかが、このフラットなリストからは読み取れません。
*   **例外系・失敗時の挙動**: 正常系のフローしか記述されておらず、例えば `ReadRegistryData` で失敗した場合のリカバリフローが不明です。

## 4. 推奨構成
1.  **初期化オーバービュー**: アプリ起動からメイン画面表示までのハイレベルなフロー図（Mermaid活用推奨）。
2.  **MFCフレームワーク階層**: アプリ・フレーム・ドキュメント・ビューの生成順序。
3.  **ルータ設定データ層**: `CDkRouter` 抽象クラスと機種別の具象クラス生成フロー。
4.  **詳細解析シーケンス**: `Analize` メソッド内部の `Proc` 呼び出し順序とその理由（データ依存関係）。
5.  **外部リソース依存**: レジストリ、設定ファイル、ルータ情報の取得タイミング。
6.  **トラブルシューティング**: 初期化失敗時のログ出力位置。

## 5. 各構成の必須記載項目
*   **ルータ種別判定と分岐**: `AnalizeRouterType` の判定ロジック詳細。
*   **コンストラクタチェイン**: 基底クラスから派生クラスへの初期化順序。
*   **プロトコル解析順序の正当性**: なぜ `ProcIP` -> `ProcPPPOE` -> `ProcNAT` の順でなければならないかの理由（IP確定待ち等）。

## 6. 将来の担当者向け補強提案
*   **クラス図の併記**: `CDkRouter` 継承ツリーの最新化。
*   **用語定義の追加**: `HeData`、`Procid` 等のドメイン用語解説。
*   **シーケンス図の自動生成**: ログからPlantUML等を生成するツールの導入。

## 7. 欠落情報一覧（要確認事項）
*   [ ] **機種判定の分岐ロジック**: `CDkRouter830` や `CDkRouter1220` の使い分け条件。
*   [ ] **解析中断条件**: `ProcXXX` が失敗した際、後続のプロトコル解析を続行するか否か。
*   [ ] **非同期処理の有無**: `ReadTftpFile` 等の通信処理がメインスレッドで行われているか。

## 8. 超要約版（最短まとめ）
*   **現状**: 起動～詳細解析（`Analize`）までのフラットな呼出履歴。
*   **課題**: UI初期化とデータ解析が混在し、機種依存や解析順序の意図が不明瞭。
*   **対策**: MFCライフサイクル、機種生成、詳細解析（`Proc`系）の3層に構造化して記述する。
*   **提案**: 詳細解析におけるプロトコル間のデータ依存関係（順序の必然性）を明文化する。
