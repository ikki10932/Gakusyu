# 機種毎ルータconfig読み込み処理のレビュー報告

本資料は、4種類のルータ（RTX1220 / 830 / 1200 / 1000）における設定ファイル（config）読み込みフローを比較し、共通基盤と機種固有実装の構造を長期保守性の観点からレビューしたものです。

## 1. 改善の方向性
現在、RTX1220/830/1200の3機種は `CDkRouter1200` を中間基底クラスとして共通利用していますが、RTX1000のみ `CDkRouter1000` が直接実装を担う「二系統の継承ツリー」が混在しています。今後は、これらを統合する「共通解析エンジン（Generic Parser）」を定義し、機種間での解析ルールの漏れを物理的に防ぐ設計への移行を推奨します。

## 2. 現行資料の問題点（長期保守観点）
*   **機種間の「隠れた差異」**: RTX1200系列とRTX1000で、`ProcISDN` や `ProcPPPOE` の位置・有無が微妙に異なります。これが意図的な「仕様差」なのか、実装上の「時期のズレ」なのかが資料から判別できません。
*   **中間基底クラスへの過度な依存**: `CDkRouter1200` が RTX1220/830 の具象クラスの役割を代替しているように見えます。最新機種（RTX1220）専用の解析ロジックを追加する際、既存機種（1200系）に影響が出やすい構造です。
*   **カラーリングによる依存把握の限界**: 現行資料では赤字などで差異を表現していますが、文字ベースのドキュメントに落ちた際にこの情報の重要度が欠落するリスクがあります。

## 3. 推奨章立て構成
1.  **解析基盤の継承構造**: `CDkRouter` -> `CDkRouter1200` 系と `CDkRouter1000` 系のクラス構成図。
2.  **共通解析フロー**: すべての機種で必ず実行される `ReadTftpFile` から `Analize` までの共通シーケンス。
3.  **機種別の特殊解析ルール**:
    *   RTX1200系：ISDN/PP解析の優先度。
    *   RTX1000系：独自クラス（CDkRouter1000）によるオーバーライド項目。
4.  **プロトコル別プロセッサ一覧**: `ProcNAT`, `ProcDHCP`, `ProcDNS` 等の対応状況マトリクス。
5.  **将来の拡張ガイド**: 新機種追加時に `CDkRouter1200` を継承すべきか、新規クラスを作成すべきかの判断基準。

## 4. 各章の必須記載項目
*   **`AnalizeRouterType` の位置づけ**: 読み込んだファイルの解析ルールを決定する「Strategyパターンの選択」としての役割明示。
*   **`HeDataLoad` 後のデータ同期**: 解析結果がUI（Grid）へ反映される `HeGridSet` までの連動フロー。
*   **`ConvertIpDelmit` の分布**: 機種によって呼ばれる位置が異なる理由（ファイルフォーマットの微差）の詳細。

## 5. 将来の担当者向け補強提案
*   **対応プロトコル対照表の作成**: 機種ごとにどの `ProcXXX` が呼ばれるかを一覧化したマトリクスをドキュメント化してください。
*   **解析例外ログの共通化**: `ProcXXX` 内で解析エラーが起きた際、どの機種でも同じフォーマットでログ出力される仕組みを導入すべきです。

## 6. 欠落情報一覧（要確認事項）
*   [ ] **RTX1200系の中間基底利用の理由**: なぜRTX1220が専用クラスを持たず、1200の関数を直接叩いているのか（実装の共通化レベルの確認）。
*   [ ] **ISDN処理の将来性**: RTX1000や1200に残る `ProcISDN` が最新機種で不要となった場合のクリーンアップ方針。
*   [ ] **`GetProviderNo` の取得先**: 設定ファイル内のどのセクションに基づいているかの仕様明記。
